- 食物语（FoodGPT）项目需求文档
  ## 1. 项目概述
  ### 1.1 项目名称
  **食物语（FoodGPT）**
  ### 1.2 项目目标
  1. 提供基于身体数据的个性化饮食建议
  2. 通过数据追踪帮助用户了解自身营养状况
  3. 利用 AI 技术生成智能食谱推荐
  4. 针对女性用户提供专属周期管理和饮食指导
  5. 构建"记录→分析→推荐→干预"的完整健康管理闭环
  ### 1.3 用户人群
  - **核心用户**：关注健康饮食的女性用户（大学生、职场女性为主）
  - **用户需求**：减脂、增肌或健康维持
  - **年龄范围**：18-35岁
  ### 1.4 产品形态
  - **应用类型**：桌面小程序
  - **界面风格**：简洁、直观
  - **功能模块**：用户管理、身体数据管理、营养素追踪、智能食谱推荐、周期管理、智能干预提醒等
  ***
  ## 2. 功能需求
  项目分为两大层次：**基础功能层（全民通用）** 和 **女性友好专区（增强层）**。
  ### 2.1 基础功能层
  #### 2.1.1 用户管理与身体数据管理（B-02）

| 项目 | 内容 |
|------|------|
| 功能名称 | 用户管理与身体数据管理 |
| 功能描述 | 用户录入身高、体重、年龄、活动量，系统自动计算 BMI 和 BMR，支持后续修改更新 |
| UI 布局描述 | 位于主页仪表盘顶部"我的身体数据"卡片（圆角16px，浅粉色渐变背景，内边距24px），左右两栏布局：左栏：四个输入控件纵向排列（身高Spinner带"cm"单位、体重Spinner带"kg"单位、年龄Spinner带"岁"单位、活动量ComboBox含4个选项），每个带圆角边框；右栏：BMI圆形进度指示器（中心显示数值，外圈环形进度条）+ BMR数值显示卡（大号数字+"基础代谢率 kcal/天"）+ 推荐热量范围条；底部："保存数据"（粉色主按钮）和"修改数据"（边框按钮）并排右对齐 |
| 前置条件 | 用户已打开应用进入主界面 |
| 后置结果 | 数据保存成功，BMI/BMR显示更新，数据同步至热量目标计算 |
| 操作步骤 | 输入身高→输入体重→输入年龄→选择活动量→点击"保存"→系统自动计算BMI/BMR |
| 验收标准 | 身高、体重、年龄输入框校验有效范围（身高 100-250cm，体重 30-200kg，年龄 10-100岁）；活动量选择项包含：久坐、轻度、中度、活跃；BMI 计算公式正确：体重(kg) / 身高(m)²；BMR 计算公式正确（采用 Mifflin-St Jeor 公式）；数据修改后实时更新 BMI 和 BMR 显示 |

#### 2.1.2 体重追踪（B-03）

**功能描述**：

- 用户可定期记录体重
- 系统以折线图展示体重变化趋势

**用户场景**：

- 用户每周固定时间记录体重，观察变化趋势
- 用户在减脂/增肌过程中，需要查看体重变化曲线

**验收标准**：

- 支持手动录入日期和体重
- 折线图展示至少最近 30 天的体重数据
- 图表支持缩放和拖拽查看
- 体重记录支持删除和修改

**UI布局**：

- 位于主页仪表盘中部"体重趋势"卡片（圆角16px，白色背景，带阴影）：顶部：标题 + 时间范围切换按钮组（7天/30天/90天，选中粉色填充）+ "＋添加记录"浮动按钮；图表区：JavaFX LineChart，X轴日期，Y轴体重值，数据点带圆点标记和数值标签，悬停显示详情；底部：最近3条记录列表（日期/体重/编辑/删除图标）

#### 2.1.3 菜谱管理（B-04）

**功能描述**：

- 自动记录联网搜索时用户选择保存的菜谱（包含名称、食材、烹饪步骤、营养成分）
- 支持为保存的菜谱设置分类标签
- 支持查看和删除已保存的菜谱

**用户场景**：

- 用户从网络搜索到心仪的菜谱，一键保存到本地
- 用户为保存的菜谱设置分类（早餐/午餐/晚餐/加餐/零食）
- 用户查看已保存菜谱的详细信息和烹饪步骤
- 用户删除不再需要的菜谱

**验收标准**：

- 联网搜索的菜谱可自动保存到本地数据库，包含名称、食材、烹饪步骤、营养成分
- 支持为菜谱设置分类标签（早餐/午餐/晚餐/加餐/零食）
- 支持按名称、分类搜索菜谱
- 菜谱列表支持分页展示
- 支持删除已保存的菜谱

**UI布局**：

- 位于"菜谱管理"独立标签页：顶部工具栏：搜索框（带放大镜图标）+ 分类筛选下拉框（全部/早餐/午餐/晚餐/加餐/零食）+ "＋新建菜谱"按钮（粉色）；中部：菜谱卡片网格（每行3张，gap16px），每张卡片（圆角12px，白色背景，悬停上浮阴影）显示图片占位区、菜名（粗体）、分类标签（彩色小圆角）、食材前3项（灰色小字）、热量值（右上角）、底部"查看详情"和"删除"按钮；底部：分页控件

#### 2.1.4 联网搜索菜谱（B-05）

| 项目 | 内容 |
|------|------|
| 功能名称 | 联网搜索菜谱 |
| 功能描述 | 用户输入已有食材，调用搜索API推荐菜谱，选择后自动保存到本地 |
| UI 布局描述 | 位于"联网搜索"独立标签页：顶部：搜索输入框（提示"输入食材，如：番茄 鸡蛋"）+ 粉色搜索按钮；偏好提示区："已应用偏好：[标签]"标签行；中部：搜索结果卡片纵向排列，每张卡片（圆角12px，白色背景）显示图片（80×80px）、菜名（粗体）、食材标签、简介摘要（两行截断）、热量值、右下角"保存到本地"按钮（保存后变"已保存"灰色）；底部：加载动画/结果数量/错误提示+重试按钮 |
| 前置条件 | 已连接网络，已配置搜索API密钥 |
| 后置结果 | 菜谱保存到本地数据库，显示在菜谱管理列表中 |
| 操作步骤 | 输入食材→点击搜索→浏览结果→点击"保存到本地" |
| 验收标准 | ①支持多食材（空格分隔）②展示名称/食材/简介/热量 ③保存后按钮变"已保存" ④网络异常显示友好提示+重试 |

#### 2.1.5 联网搜索增强（B-06）

**功能描述**：

- 根据用户偏好（低脂/高蛋白/素食/无糖等）自动优化搜索关键词

**用户场景**：

- 用户偏好低脂饮食，搜索时自动追加"低脂"关键词
- 用户是素食者，搜索结果自动过滤含肉类的菜谱

**验收标准**：

- 用户可设置饮食偏好标签（低脂、高蛋白、素食、无糖、低碳水）
- 搜索时自动将偏好标签追加到搜索请求中
- 搜索结果优先展示符合用户偏好的菜谱

**UI布局**：

- 设置面板："饮食偏好"卡片，五个圆角开关按钮（低脂/高蛋白/素食/无糖/低碳水），启用为粉色填充，未启用为灰色边框；搜索界面：搜索框下方显示"已应用偏好：[标签]"标签行，每个标签带×可临时移除；结果顶部：显示"已根据您的偏好过滤结果"提示条（3秒淡出）

#### 2.1.6 用餐记录（B-08）

| 项目 | 内容 |
|------|------|
| 功能名称 | 用餐记录 |
| 功能描述 | 记录每日饮食（菜品、份量、餐次），系统自动归档历史记录 |
| UI 布局描述 | 位于"用餐记录"独立标签页：顶部：日期选择器（左右箭头+日期显示）+ "＋添加记录"粉色按钮；餐次切换：四个标签（早餐/午餐/晚餐/加餐），选中粉色背景白字；记录列表：当前餐次记录卡片（浅色背景，圆角10px），显示菜品名、份量+单位、营养素小图标+数值，右侧编辑/删除图标；空状态显示"还没有记录"；底部：半透明汇总条"今日摄入：XXkcal｜蛋白质XXg｜碳水XXg｜脂肪XXg" |
| 前置条件 | 本地菜谱库中已有菜谱可供选择 |
| 后置结果 | 用餐记录保存成功，列表刷新，营养素数据同步更新 |
| 操作步骤 | 选择日期→点击餐次标签→点击"添加记录"→选择菜品→输入份量→确认保存 |
| 验收标准 | ①支持4个餐次 ②菜品下拉选择 ③份量支持份/克/毫升 ④显示最近7天记录 ⑤支持编辑删除 ⑥底部实时汇总 |

#### 2.1.7 营养素分析（B-09）

**功能描述**：
- 根据用餐记录自动计算每餐及全天的蛋白质/碳水/脂肪摄入量
- 按日期汇总展示

**用户场景**：
- 用户想知道今天摄入了多少蛋白质和碳水
- 用户需要查看某一天的营养素摄入明细

**验收标准**：

- 自动计算每餐的蛋白质、碳水、脂肪摄入量
- 汇总计算全天的营养素摄入总量
- 按日期展示历史营养素摄入数据
- 数据精度保留一位小数

**UI布局**：
- 位于"营养素分析"独立标签页：顶部：日期选择器（左右箭头）+ "刷新"按钮；图表区：分组柱状图（X轴餐次，Y轴克数，三色柱状：粉/橙/黄），顶部显示数值标签；数据卡片：三大营养素横向卡片（flex:1，圆角12px），显示名称、摄入总量、目标值、进度小条；底部：可折叠明细表，按餐次展开，每行显示菜品/份量/三大营养素数值

#### 2.1.8 营养素进度（B-10）

**功能描述**：

- 以进度条形式展示当日三大营养素摄入情况（已摄入/目标值）

**用户场景**：

- 用户在主页查看今日营养摄入进度
- 用户想知道还需要摄入多少蛋白质才能达到目标

**验收标准**：

- 展示蛋白质、碳水、脂肪三个进度条
- 每个进度条显示已摄入量/目标值
- 进度条颜色根据完成度变化（未达标：橙色，达标：绿色）
- 实时更新进度数据

**UI布局**：
- 位于主页仪表盘"今日营养进度"卡片（圆角16px，白色背景）：三个进度条纵向排列（间距16px），每个显示：图标+名称、进度条（高度12px，圆角6px，背景浅灰，填充色渐变：蛋白质粉色/碳水橙色/脂肪黄色）、右侧数值"已摄入/目标值g"；下方显示完成度总结文字（如"蛋白质已完成80%"）

#### 2.1.9 均衡度判断（B-11）

**功能描述**：

- 分析当日营养结构是否均衡，生成 0-100 的综合评分

**用户场景**：

- 用户想了解今日饮食是否均衡
- 用户根据评分调整后续饮食计划

**验收标准**：

- 根据三大营养素的摄入比例计算均衡度评分
- 评分范围 0-100，60 分以下为不均衡，80 分以上为优秀
- 显示评分等级和简要评语
- 点击查看详细分析报告

**UI布局**：
- 位于主页仪表盘"均衡度评分"卡片（圆角16px，白色背景）：左侧：圆形进度指示器（直径80px，外圈环形进度条，中心显示评分数字）；右侧：等级标签（优秀/良好/需改善，彩色文字）+ 评语（灰色小字）；底部："查看详细分析"文字按钮（居中，灰色带箭头）



#### 2.1.10 主动干预（B-12）

**功能描述**：

- 每天自动生成营养报告
- 针对营养缺口给出具体补充建议（如"蛋白质不足，建议晚餐补充鸡胸肉"）

**用户场景**：

- 用户打开应用时收到当日营养报告
- 用户收到针对性的营养补充建议

**验收标准**：

- 每日首次打开应用时自动显示营养报告
- 报告包含当日摄入总结、均衡度评分、营养缺口分析
- 根据缺口生成具体的食材建议
- 建议内容具体可操作，包含食材名称和推荐摄入量

**UI布局**：
- 模态弹窗（居中，宽480px，圆角20px，白色背景带阴影）：顶部："今日营养报告"标题+日期+关闭按钮；中部：摄入总结（三大营养素数值+进度小条）+ 均衡度评分 + 缺口分析列表（营养素名+缺口比例红色进度条）；底部：浅粉色卡片"补充建议"+具体建议文字；按钮："我知道了"粉色主按钮居中

### 2.2 女性友好专区（增强层）

专区复用基础功能的身体数据、用餐记录和营养素分析结果，在此基础上增加以下功能：

#### 2.2.1 健康目标设定（F-01）

**功能描述**：

- 用户可选择目标：减脂 / 增肌/增重 / 维持健康

**用户场景**：

- 用户开始使用时设定健康目标
- 用户根据自身情况调整目标

**验收标准**：

- 目标选项包含：减脂、增肌/增重、维持健康
- 选择目标后系统自动调整推荐热量范围
- 支持随时修改目标

**UI布局**：
- 位于"女性专区"面板顶部（圆角16px，浅粉色渐变背景）：标题行："我的健康目标"+当前目标标签；目标卡片：三个横向卡片（减脂/增肌/维持），选中为粉色边框+阴影+右上角"✓"标记；底部：提示条"当前目标将自动调整热量推荐"+ "保存目标"粉色按钮

#### 2.2.2 生理周期管理（F-02）

**功能描述**：

- 用户记录月经周期起止日期
- 系统自动识别当前周期阶段，动态调整推荐策略（如经期推荐补铁食谱）

**用户场景**：

- 用户记录本次月经开始日期
- 用户在经期收到特殊饮食建议

**验收标准**：

- 支持录入月经周期开始日期和周期长度
- 系统自动计算下次月经预测日期
- 识别当前周期阶段（经期、卵泡期、排卵期、黄体期）
- 不同周期阶段推荐不同的饮食策略

**UI布局**：
- 位于"女性专区"面板（圆角16px，白色背景）：顶部："生理周期"+ "＋添加记录"按钮；阶段指示器：四个阶段横向展示（经期/卵泡期/排卵期/黄体期），当前阶段高亮；日历区：月视图，周期日期标记特殊颜色；底部：上次周期/预测下次日期信息
#### 2.2.3 女生专属食谱库（F-03）

**功能描述**：

- 内置五类食谱：减脂、增肌、健康维持、经期调理、孕期调理

**用户场景**：

- 用户在经期想查看适合的调理食谱
- 用户准备怀孕或怀孕期间需要特殊饮食指导

**验收标准**：

- 食谱库包含五类：减脂、增肌、健康维持、经期调理、孕期调理
- 每类至少包含 20 道精选食谱
- 食谱标注适用周期阶段
- 支持按类别筛选和搜索

**UI布局**：
- 位于"女性专区"→"专属食谱"：顶部：五个分类标签（减脂/增肌/健康维持/经期调理/孕期调理），选中粉色填充；中部：食谱卡片列表（图片/名称/分类标签/适用阶段/热量），右下角"查看详情"和"保存到本地"按钮；底部：搜索框（按名称或食材搜索）

#### 2.2.4 热量目标计算（F-04）

**功能描述**：

- 基于 BMR × 活动系数 + 目标调整值，自动计算每日推荐热量范围

**用户场景**：

- 用户设定目标后，系统自动计算每日热量需求
- 用户活动量变化时，热量目标自动更新

**验收标准**：

- 根据 BMR × 活动系数计算基础热量需求
- 根据目标类型调整热量（减脂：-300\~500kcal，增肌：+300\~500kcal）
- 显示每日推荐热量范围
- 身体数据或活动量变化时自动重新计算

**UI布局**：
- 位于"女性专区"面板顶部"热量目标"区域：大号数字显示"1800-2100 kcal"；下方显示计算依据（BMR值 + 活动系数 + 目标调整值）；"查看详细计算"文字按钮

#### 2.2.5 食物偏好录入（F-05）

**功能描述**：

- 用户可设置偏爱食材和忌口食材

**用户场景**：

- 用户不喜欢吃香菜，设置为忌口
- 用户喜欢吃海鲜，设置为偏爱

**验收标准**：

- 支持添加偏爱食材和忌口食材列表
- 偏爱食材在推荐时优先考虑
- 忌口食材在推荐时自动过滤

**UI布局**：
- 位于"设置"→"食物偏好"：偏爱区："偏爱食材"+输入框+"添加"按钮，下方标签列表（点击×移除）；分隔线；忌口区："忌口食材"+输入框+"添加"按钮，下方标签列表（点击×移除）

#### 2.2.6 专属食谱匹配（F-06）

**功能描述**：

- 从菜谱库中过滤出符合热量目标 + 偏好 + 忌口的菜谱

**用户场景**：

- 用户想找到既符合减脂目标又不含忌口食材的菜谱
- 用户根据当前周期阶段获取专属推荐

**验收标准**：

- 过滤条件包含：热量范围、偏爱食材、忌口食材、周期阶段
- 推荐结果按匹配度排序
- 显示匹配度评分
- 支持保存匹配结果到收藏

**UI布局**：
- 位于"女性专区"→"专属匹配"：顶部：过滤条件摘要（热量范围/偏好/忌口/周期阶段）；中部：匹配结果列表（卡片形式），每张显示菜名、匹配度百分比（进度条）、符合条件数；卡片右下角："查看"和"收藏"按钮；无结果时显示提示

#### 2.2.7 搜索关键词增强（F-07）

**功能描述**：

- 搜索时自动追加女性专属关键词 + 用户偏好

**用户场景**：

- 用户搜索菜谱时，结果自动偏向女性友好的选项
- 用户在经期搜索时自动追加"经期调理"关键词

**验收标准**：

- 根据当前周期阶段自动追加相关关键词
- 根据用户偏好追加偏好关键词
- 搜索结果优先展示符合条件的菜谱

**UI布局**：
- 位于"联网搜索"面板搜索框下方：提示区域显示"本次搜索已应用关键词：[标签]"，关键词以标签形式展示，可点击×移除

#### 2.2.8 营养缺口分析（F-08）

**功能描述**：

- 对比用户目标与当日实际摄入，识别缺少的营养素

**用户场景**：

- 用户想知道今天哪些营养素摄入不足
- 用户根据缺口调整明天的饮食计划

**验收标准**：

- 对比每日目标摄入量与实际摄入量
- 识别缺口超过 20% 的营养素
- 展示缺口比例和具体数值
- 按缺口严重程度排序

**UI布局**：
- 位于"营养素分析"→"缺口分析"：顶部："营养缺口分析"+分析时间；中部：缺口营养素卡片列表（按缺口比例从高到低排列），每张显示营养素名、实际摄入量/目标量、缺口百分比红色进度条；底部："生成补充建议"按钮（跳转F-09）
#### 2.2.9 行动建议生成（F-09）

**功能描述**：

- 根据营养缺口生成具体的食材建议（如"缺铁，建议吃菠菜/红肉"）

**用户场景**：

- 用户收到针对性的营养补充建议
- 用户根据建议调整餐食选择

**验收标准**：

- 针对每种缺口营养素生成至少 3 种食材建议
- 建议考虑用户的饮食偏好和忌口
- 建议食材常见易获取
- 显示每种食材的营养成分含量

**UI布局**：
- 位于"营养素分析"→"补充建议"：每个缺口对应一个建议卡片，显示营养素名称、推荐食材列表（至少3种，带✓或标记是否匹配偏好）、每种食材的营养成分含量；底部："采纳建议"按钮（加入购物清单）

## 5. 约束与假设

### 5.1 技术约束

- 采用 JavaFX 框架开发，确保跨平台兼容性
- 使用 MyBatis-Plus + SQLite（轻量级嵌入式数据库） 数据库存储数据
- 联网功能依赖第三方菜谱搜索 API
- AI 咨询功能依赖 DeepSeek 大模型 API

### 5.2 数据约束

- 用户身体数据需符合合理范围（身高 100-250cm，体重 30-200kg）
- 菜谱营养成分数据需准确录入
- 生理周期数据由用户手动录入，系统不进行医疗判断
- 联网搜索结果质量依赖第三方 API 返回数据

### 5.3 功能约束

- AI 建议仅供参考，不能替代专业医疗建议
- 孕期调理食谱仅供参考，用户应咨询医生意见
- 网络异常时，搜索和 AI 功能不可用，但不影响本地数据操作

### 5.4 假设条件

- 用户具备基本的计算机操作能力
- 用户拥有稳定的网络连接（用于搜索和 AI 功能）
- 用户愿意定期录入身体数据和饮食记录
- 用户了解应用提供的是健康建议而非医疗诊断
